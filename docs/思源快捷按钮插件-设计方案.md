## 1. 结论

该插件想法**可行**，但必须把需求拆成两层：

1. 稳定可交付层：
   - 自定义新增插件按钮
   - 在顶栏、左/右/底部 Dock、底栏状态区展示插件自己的快捷按钮
   - 提供可视化配置界面、图标选择、命令绑定、位置预览、持久化存储
   - 快捷调用一组“稳定可调用”的思源内置命令和插件自定义命令
2. 实验增强层：
   - 调整思源原生按钮的显隐、顺序、图标
   - 绑定全部命令面板内置命令

如果把需求理解为“稳定控制思源全部原生按钮和全部命令面板命令”，则**不完全可行**，因为公开插件 API 没有提供完整的原生工具栏编排 API，也没有提供完整的命令面板命令注册表/执行器。

## 2. 可行性拆解

|需求点|结论|说明|
|---|---|---|
|自定义配置工具栏按钮|可行|公开 API 已提供 `addTopBar`、`addStatusBar`、`addDock`、`saveData/loadData`，足以实现插件自有按钮系统|
|顶栏按钮是否显示、显示位置、图标|部分可行|插件自有按钮可稳定控制；原生顶栏按钮只能做实验性 DOM 适配|
|左侧/右侧/底部按钮是否显示、显示位置、图标|部分可行|插件 Dock 按钮可稳定控制；原生 Dock 按钮重排/换图标依赖内部布局结构|
|快捷调用命令面板中的命令|部分可行|公开 `globalCommand` 能稳定触发一部分全局命令；并非全部命令面板项都能通过公开 API 调用|
|可视化配置界面|可行|模板本身基于 Vue，适合实现复杂设置页|
|工具栏位置预览|可行|完全由插件前端自行渲染即可|
|内置按钮图标选择|可行|可基于思源内置 SVG id、自定义 SVG、Emoji 构建图标选择器|
|命令面板内置命令关联选择|部分可行|应做“受支持命令目录”，不要承诺覆盖全部上下文命令|

## 3. 关键证据

### 3.1 公开插件能力足够支撑插件自有按钮系统

从 `app/src/plugin/index.ts` 与 `petal/siyuan.d.ts` 可以确认：

- `addTopBar()`：新增顶栏按钮
- `addStatusBar()`：新增底栏状态区元素
- `addDock()`：新增 Dock 入口，并支持 `position`、`icon`、`index`、`show`
- `addCommand()`：新增插件命令，自动进入命令面板的插件命令区
- `openSetting()`、`Setting`：可接入设置界面
- `saveData()/loadData()/removeData()`：可持久化插件配置
- `globalCommand()`：公开暴露了一部分内置全局命令执行入口
- `openEmoji()`：可复用思源图标/Emoji 面板能力

其中 `IPluginDockTab.position` 明确支持：

- `LeftTop`
- `LeftBottom`
- `RightTop`
- `RightBottom`
- `BottomLeft`
- `BottomRight`

这说明“插件自有左/右/底部按钮”不是猜测，而是官方已有明确支撑。

### 3.2 原生顶栏和底栏是思源内部自己维护的

从 `app/src/layout/topBar.ts` 与 `app/src/layout/status.ts` 可确认：

- 思源原生顶栏按钮是内部硬编码渲染的，如 `barWorkspace`、`barSync`、`barCommand`、`barSearch`、`barMode`
- 思源原生底栏状态区也是内部初始化的，如 `barDock`、`statusHelp`
- 公开插件 API 只允许“追加插件元素”，没有公开“重排原生按钮”“替换原生图标”“统一读取所有原生按钮元数据”的接口

因此：

- 原生按钮显隐、顺序、图标替换只能走 DOM/CSS/内部对象适配
- 这类能力必须标注为实验特性，并带版本保护和降级逻辑

### 3.3 命令面板并没有公开“全量命令注册表”

从 `app/src/boot/globalEvent/command/panel.ts` 可确认：

- 命令面板内置命令是通过硬编码白名单从 `window.siyuan.config.keymap.general` 和 `window.siyuan.config.keymap.editor.general` 组装出来的
- 插件命令来自 `app.plugins[].commands`
- 命令执行链优先走 `globalCommand()`，然后再走内部的 `execByCommand()`

这意味着：

1. “插件命令”可以稳定列出和调用
2. “一部分内置全局命令”可以稳定列出和调用
3. “全部上下文命令面板项”没有公开统一入口

所以命令绑定设计必须采用“维护一份受支持命令目录”的思路，而不是尝试动态抓取全部命令面板项并承诺长期兼容。

## 4. 推荐的产品边界

## 4.1 V1 稳定版边界

V1 推荐只承诺以下能力：

- 管理插件自有快捷按钮
- 支持这些按钮渲染到：
  - 顶栏
  - 左侧 Dock
  - 右侧 Dock
  - 底部 Dock
  - 底栏状态区
- 支持配置：
  - 显示/隐藏
  - 所在区域
  - 区域内位置
  - 图标
  - 标题
  - 点击动作
- 支持绑定动作：
  - 公开支持的内置全局命令
  - 插件自身注册命令
  - 插件封装动作，如打开设置、打开某个自定义页签、打开 URL、调用指定内核 API
- 提供可视化配置界面：
  - 布局预览
  - 图标选择
  - 命令选择
  - 拖拽排序
  - 导入/导出配置

## 4.2 V1 不建议承诺

- 稳定控制思源原生顶栏所有按钮
- 稳定控制思源原生底栏所有按钮
- 稳定更换原生 Dock 按钮图标
- 稳定调用全部命令面板上下文命令
- 首版同时完整支持桌面端、浏览器端、移动端

首版建议聚焦：

- 桌面端
- 非窗口模式
- SiYuan `v3.5.7`

## 4.3 V1.5 或实验特性

实验层可以做，但必须默认关闭：

- 原生顶栏按钮显隐控制
- 原生 Dock 按钮显隐控制
- 原生按钮顺序微调
- 非公开命令调用适配器

这类能力需要：

- 版本白名单
- 失败即静默降级
- 在 UI 中明确标注“实验功能”

## 5. 方案设计

## 5.1 总体架构

建议把插件拆成 6 个核心模块：

1. `PluginBootstrap`
   - 插件生命周期入口
   - 加载配置
   - 注册命令
   - 在 `onLayoutReady()` 后挂载按钮
2. `ConfigStore`
   - 配置读写
   - 版本迁移
   - 默认值合并
3. `CommandRegistry`
   - 维护“受支持命令目录”
   - 统一命令执行
   - 对内区分稳定命令与实验命令
4. `SurfaceRenderer`
   - 按区域渲染按钮
   - 顶栏、状态栏、Dock 分别适配
   - 保持渲染销毁幂等
5. `SettingsUI`
   - Vue 配置界面
   - 区域预览、图标选择、命令选择、排序
6. `CompatibilityAdapter`
   - 只承接实验特性
   - DOM 查询、MutationObserver、版本判断、降级处理

## 5.2 数据模型

建议配置结构如下：

```ts
type SurfaceType =
  | "topbar"
  | "statusbar-left"
  | "statusbar-right"
  | "dock-left-top"
  | "dock-left-bottom"
  | "dock-right-top"
  | "dock-right-bottom"
  | "dock-bottom-left"
  | "dock-bottom-right";

type ActionType =
  | "builtin-global-command"
  | "plugin-command"
  | "custom-action"
  | "open-url"
  | "experimental-internal-command";

interface PowerButtonItem {
  id: string;
  title: string;
  visible: boolean;
  iconType: "builtin" | "emoji" | "svg";
  iconValue: string;
  surface: SurfaceType;
  order: number;
  actionType: ActionType;
  actionId: string;
  tooltip?: string;
  confirm?: {
    enabled: boolean;
    title?: string;
    content?: string;
  };
}

interface PowerButtonsConfig {
  version: 1;
  desktopOnly: boolean;
  items: PowerButtonItem[];
  experimental: {
    nativeToolbarControl: boolean;
    internalCommandAdapter: boolean;
  };
}
```

设计要点：

- `surface` 用统一枚举，避免每个区域写一套状态字段
- `actionType + actionId` 支持命令绑定扩展
- `version` 必须从首版就存在，后续方便迁移
- `experimental` 单独隔离，避免实验功能污染稳定配置

## 5.3 命令系统设计

建议将可绑定动作分为 4 类：

### A. 稳定内置全局命令

通过公开 `globalCommand()` 执行。

建议首版支持如下代表性命令：

- `fileTree`
- `outline`
- `bookmark`
- `tag`
- `inbox`
- `backlinks`
- `graphView`
- `globalGraph`
- `config`
- `globalSearch`
- `stickSearch`
- `goBack`
- `goForward`
- `recentDocs`
- `mainMenu`
- `dailyNote`
- `newFile`
- `riffCard`
- `syncNow`

这些命令可直接做成“内置命令目录”，绑定最稳定。

### B. 插件自定义命令

通过 `addCommand()` 注册。

用途：

- 复用统一命令执行逻辑
- 让按钮点击与命令面板执行走同一条代码路径
- 为未来快捷键支持留统一入口

### C. 插件封装动作

这类动作不强依赖命令面板，但能满足用户操作目标：

- 打开插件设置
- 打开自定义 Dock 页签
- 打开指定文档/自定义页签
- 打开链接
- 调用封装过的内核 API

### D. 实验性内部命令

仅在开启实验模式时可选。

来源于 `command/panel.ts` 里内部 `execByCommand()` 处理的命令，如：

- `search`
- `replace`
- `addToDatabase`
- `move`
- 部分编辑器上下文命令

这类命令的问题是：

- 不属于公开插件 API
- 依赖当前焦点、编辑器上下文、文件树上下文
- 不同版本容易变化

因此只能作为后续增强，不建议进 V1 默认功能。

## 5.4 按钮渲染策略

### 顶栏

使用 `addTopBar()` 渲染插件按钮。

注意点：

- 顶栏更适合少量高频按钮
- 公开 API 只支持追加按钮，不适合做“全面接管顶栏”
- 首版应只管理插件自有顶栏按钮

### 底栏状态区

使用 `addStatusBar()` 渲染。

适合：

- 一键动作
- 状态切换
- 轻量入口

### 左/右/底部 Dock

使用 `addDock()` 渲染。

这是本插件最适合承载“快捷按钮集合”的区域，因为：

- 官方明确支持位置、图标、显示状态、顺序
- 能天然承载复杂内容页和按钮入口
- 对原生顶栏侵入最小

建议把“主按钮配置中心”放在一个自定义 Dock 中，同时再允许用户把常用按钮同步渲染到顶栏或状态栏。

## 5.5 配置界面设计

建议设置界面采用 3 栏布局：

### 左侧：区域预览

- 顶栏预览
- 左 Dock 预览
- 右 Dock 预览
- 底部 Dock 预览
- 状态栏预览

用途：

- 用户一眼看到按钮会出现在哪里
- 支持拖拽排序

### 中间：按钮列表

- 新建按钮
- 复制按钮
- 删除按钮
- 显示/隐藏
- 当前绑定区域

### 右侧：属性编辑器

- 标题
- 图标
- 目标区域
- 区域内顺序
- 命令来源
- 命令选择
- 提示文案
- 高级选项

图标选择器建议支持：

- 思源内置图标 id
- Emoji
- 自定义 SVG

## 5.6 生命周期设计

建议流程：

1. `onload()`
   - 读取配置
   - 初始化命令目录
   - 注册插件命令
   - 准备设置界面数据
2. `onLayoutReady()`
   - 渲染顶栏按钮
   - 渲染状态栏按钮
   - 渲染 Dock 按钮
3. 配置变更
   - 更新内存配置
   - `saveData()`
   - 局部重渲染
4. `onunload()`
   - 销毁 DOM 引用
   - 解绑事件
   - 清理观察器

建议把按钮渲染放在 `onLayoutReady()` 后执行，不要继续沿用旧示例里在 `onload()` 直接挂顶栏/状态栏的写法。

## 6. 建议的实现边界

## 6.1 MVP

MVP 建议定义为：

- 可创建多个快捷按钮
- 可把按钮放到顶栏、状态栏、Dock
- 可选择图标
- 可绑定稳定命令
- 可拖拽排序
- 可持久化
- 可导入/导出 JSON

## 6.2 后续增强

- 实验性控制原生工具栏按钮
- 导入官方命令目录更新
- 配置模板预设
- 分工作空间配置
- 条件显示规则

## 7. 风险与应对

|风险|等级|说明|应对|
|---|---|---|---|
|原生工具栏 DOM 结构变动|高|影响实验性原生按钮控制|默认关闭实验能力，版本校验，失败降级|
|命令面板内部执行逻辑变化|高|影响非公开命令绑定|V1 只支持 `globalCommand` 目录|
|顶栏拥挤导致按钮折叠|中|过多按钮会被顶栏收纳|提示用户优先使用 Dock 和状态栏|
|多端差异|中|移动端与桌面端结构差异大|首版桌面端优先|
|配置结构后续演进|中|后续字段变化可能破坏旧配置|首版即引入 `version` 与迁移器|

## 8. 最终建议

建议按“稳定核心 + 实验增强”推进：

- 把产品定义为“插件自有快捷按钮编排系统”
- 不把首版目标定义为“全局替代思源原生工具栏”
- 把命令绑定设计成“受支持命令目录”
- 首版优先桌面端和稳定 API

按这个边界推进，项目是值得做的，并且能在公开 API 范围内交付一个实用版本。
