## 1. 最新结论

基于当前实现、官方 API 边界、以及对两个社区插件的补充分析，项目后续方向应明确为：

1. 继续保持 V1 已有的“稳定命令执行层”作为主产品能力。
2. 不把“快捷键模拟”升级为默认主路径。
3. 如果要支持更多上下文命令，应新增“实验执行层”，并拆成两个独立能力：
   - `experimental-shortcut`
   - `experimental-click-sequence`
4. 原生工具栏接管、原生按钮重排、原生按钮换图标仍然只能算实验能力，且优先级低于前两项。

也就是说，项目定位不是“接管思源全部原生工具栏”，而是：

- 提供一套稳定的插件自有快捷按钮系统
- 在稳定 API 能覆盖的范围内优先使用稳定命令执行
- 对必须依赖焦点、快捷键、DOM 自动化的场景，提供显式标记的实验能力

## 2. 当前实现与后续方向

### 2.1 当前已完成的稳定能力

当前代码中已经落地并经过测试覆盖的稳定能力包括：

- 插件自有按钮模型
- 顶栏、状态栏、Dock 渲染
- 稳定内置命令目录
- 插件命令目录
- 统一执行器
- 可视化设置界面
- 图标切换
- JSON 导入导出
- 配置持久化与清洗

当前命令执行链的核心思路是：

1. 优先使用 `plugin.globalCommand()`
2. 失败时对少量稳定内置命令走 DOM 点击兜底
3. 插件命令、自定义动作、URL 由统一执行器分发

这个方向是正确的，应继续保留为主路径。

### 2.2 当前明确不应改变的原则

后续新增能力时，不应破坏以下原则：

- 稳定能力与实验能力必须分层
- `builtin-global-command` 仍然是默认推荐动作类型
- 实验能力默认关闭
- 实验能力失败时必须降级，不允许影响稳定路径
- UI 中必须明确标注“实验功能”

## 3. 两个参考项目的分析结论

## 3.1 `iamqiz/siyuan-plugin-add-shortcut-to-topbar`

### 实现方式

这个插件的核心并不是调用“命令执行 API”，而是：

1. 在配置界面录制用户按下的快捷键
2. 保存 `ctrlKey/shiftKey/altKey/metaKey/key/code/keyCode`
3. 用户点击按钮时重新构造 `KeyboardEvent("keydown")`
4. 把事件派发给活动编辑器或 `window/body`

它还做了两类附加能力：

- 顶栏按钮排序/显隐
- 编辑器工具栏项目插入、排序、隐藏

### 优点

- 方案简单
- 能覆盖“本来就能被快捷键触发”的功能
- 对没有公开 API 的功能有一定命中率

### 缺点

- 它本质是“按键重放器”，不是“命令执行器”
- 无法知道当前快捷键实际对应哪个命令
- 焦点、选区、当前面板上下文不对时，成功率会明显下降
- 没有形成“稳定能力”和“实验能力”的清晰边界

### 对本项目的借鉴结论

可借鉴：

- 快捷键录制数据结构
- 向编辑器派发事件而不是只发给 `window`
- 点击按钮触发快捷键的最小实现思路

不直接借鉴：

- 将快捷键模拟作为默认执行路径
- 使用快捷键字符串替代命令语义模型

## 3.2 `HaoCeans/siyuan-toolbar-customizer`

### 实现方式

这个项目比前者更进一步，关键点有三层：

1. 先把用户输入的 `Alt+5 / Ctrl+B` 规范化为思源 `keymap` 使用的符号格式
2. 从 `window.siyuan.config.keymap` 反查快捷键对应的命令
3. 再根据命令类型决定执行方式：
   - 直接点击原生按钮
   - 直接调用部分 `protyle` 方法
   - 向编辑器派发 `KeyboardEvent`
   - 向 `window` 派发 `KeyboardEvent`

此外它还实现了：

- 点击序列自动化
- 智能元素查找
- 等待与重试
- EventBus 驱动的工具栏刷新时机

### 优点

- 比单纯快捷键重放更接近“命令语义执行”
- 能识别是否为编辑器命令
- 对选区、焦点、编辑器上下文做了专门处理
- 点击序列的等待、重试、文本匹配设计较完整

### 缺点

- 大量依赖 `window.siyuan` 内部结构
- 大量依赖 DOM 结构和私有 `protyle` 路径
- 核心逻辑集中在超大文件中，边界不清晰
- 稳定能力与实验能力耦合过深

### 对本项目的借鉴结论

强烈建议借鉴：

- `keymap -> command -> 执行策略` 这条思路
- 对编辑器命令单独处理焦点和选区
- 点击序列的智能匹配与等待重试机制
- 监听 `loaded-protyle-dynamic` / `switch-protyle` / `loaded-protyle-static`

明确不建议照搬：

- 巨型 `toolbarManager.ts` 风格
- 大量散落的 `window.siyuan.layout` 私有结构访问
- 把“内置按钮 ID 查找”“实验自动化”“稳定命令”混在一条主链里

## 3.3 综合判断

两个项目共同证明了两件事：

1. 快捷键模拟确实能补足公开 API 覆盖不到的一部分命令。
2. 但如果不先做命令语义映射，长期会变成高风险、低可维护的 DOM 黑科技。

因此本项目后续应采用：

- 稳定层：语义命令执行
- 实验层：快捷键映射执行 + 点击序列自动化

而不是：

- 所有动作都退化为 `KeyboardEvent`

## 4. 推荐产品边界

## 4.1 稳定版边界

稳定版正式承诺以下能力：

- 管理插件自有快捷按钮
- 支持这些按钮渲染到：
  - 顶栏
  - 状态栏左/右
  - 左/右/底部 Dock
- 支持配置：
  - 显示/隐藏
  - 所在区域
  - 区域内顺序
  - 图标
  - 标题
  - 提示
  - 点击动作
- 支持动作类型：
  - `builtin-global-command`
  - `plugin-command`
  - `custom-action`
  - `open-url`

## 4.2 实验版边界

实验能力只能在显式开启后使用：

- `experimental-shortcut`
- `experimental-click-sequence`
- 原生工具栏按钮显隐/顺序适配

这些能力的共同约束：

- 默认关闭
- UI 中有明确说明
- 失败时只提示，不影响主界面
- 强依赖当前 SiYuan 版本和 DOM 结构

## 4.3 不建议承诺的能力

以下能力仍不应写入稳定版承诺：

- 稳定控制全部原生顶栏按钮
- 稳定控制全部原生 Dock 按钮
- 稳定更换原生按钮图标
- 稳定调用全部命令面板上下文命令
- 移动端与桌面端完全一致的实验自动化行为

## 5. 总体架构更新

## 5.1 当前稳定架构

当前架构已经形成 4 个稳定核心模块：

1. `ConfigStore`
   - 配置读写
   - 配置清洗
   - 配置迁移入口
2. `CommandExecutor`
   - 统一动作分发
   - 稳定命令执行
3. `SurfaceManager`
   - 顶栏、状态栏、Dock 渲染
   - 渲染销毁幂等
4. `SettingsUI`
   - 按钮列表
   - 属性编辑
   - 预览
   - 导入导出

## 5.2 推荐扩展后的架构

后续建议在不破坏当前结构的前提下，扩展为：

1. `src/core/commands/catalog.ts`
   - 稳定命令目录
2. `src/core/commands/executor.ts`
   - 主执行器
3. `src/core/commands/builtin-dom.ts`
   - 稳定 DOM 兜底
4. `src/core/commands/experimental-shortcut.ts`
   - 快捷键规范化
   - keymap 反查
   - 键盘事件生成
5. `src/core/commands/editor-context.ts`
   - 活动编辑器定位
   - 选区保存/恢复
   - 编辑器命令上下文执行
6. `src/core/commands/click-sequence.ts`
   - 点击序列执行
   - 元素等待
   - 智能匹配
   - 重试
7. `src/core/compatibility/native-toolbar.ts`
   - 原生工具栏实验适配
8. `src/core/compatibility/version-guard.ts`
   - 版本白名单
   - 降级策略

## 5.3 模块职责边界

稳定模块不得依赖：

- 非公开 `window.siyuan.layout` 结构
- DOM 结构猜测
- MutationObserver 常驻扫描

实验模块可以依赖，但必须：

- 集中在 `compatibility` 或 `experimental` 目录
- 有日志前缀和失败降级
- 被配置开关显式保护

## 6. 数据模型更新建议

## 6.1 当前稳定数据模型

当前代码中的稳定动作类型为：

```ts
type ActionType =
  | "builtin-global-command"
  | "plugin-command"
  | "custom-action"
  | "open-url";
```

这适合 V1，且已经落地。

## 6.2 下一阶段建议扩展

在进入实验层时，建议将动作类型扩展为：

```ts
type ActionType =
  | "builtin-global-command"
  | "plugin-command"
  | "custom-action"
  | "open-url"
  | "experimental-shortcut"
  | "experimental-click-sequence";
```

### 推荐扩展模型

```ts
interface ExperimentalShortcutConfig {
  shortcut: string;
  sendEscapeBefore?: boolean;
  dispatchTarget?: "auto" | "active-editor" | "window" | "body";
  allowDirectWindowDispatch?: boolean;
}

interface ClickSequenceStep {
  selector: string;
  timeoutMs?: number;
  retryCount?: number;
  retryDelayMs?: number;
  delayAfterMs?: number;
}

interface ExperimentalClickSequenceConfig {
  steps: ClickSequenceStep[];
  stopOnFailure: boolean;
}

interface PowerButtonItem {
  id: string;
  title: string;
  visible: boolean;
  iconType: "builtin" | "emoji" | "svg";
  iconValue: string;
  surface: SurfaceType;
  order: number;
  actionType: ActionType;
  actionId: string;
  tooltip?: string;
  experimentalShortcut?: ExperimentalShortcutConfig;
  experimentalClickSequence?: ExperimentalClickSequenceConfig;
}

interface PowerButtonsConfig {
  version: 1 | 2;
  desktopOnly: boolean;
  items: PowerButtonItem[];
  experimental: {
    nativeToolbarControl: boolean;
    internalCommandAdapter: boolean;
    shortcutAdapter: boolean;
    clickSequenceAdapter: boolean;
  };
}
```

### 设计说明

- `actionId` 仍然保留，便于统一列表渲染和导入导出
- 实验动作的详细参数不塞进 `actionId`
- `experimentalShortcut` 与 `experimentalClickSequence` 必须只在对应类型下生效
- 配置迁移建议在引入实验动作时把 `version` 升到 `2`

## 7. 命令系统设计更新

## 7.1 稳定执行链

稳定执行链保持如下顺序：

1. `builtin-global-command`
   - 优先 `plugin.globalCommand(commandId)`
   - 失败后走稳定 DOM 兜底
2. `plugin-command`
   - 由插件注册命令表执行
3. `custom-action`
   - 由本插件封装动作处理
4. `open-url`
   - 由统一 URL 打开器处理

这条链不应被实验能力污染。

## 7.2 新增实验执行链

### A. `experimental-shortcut`

推荐执行顺序：

1. 获取用户录入快捷键，例如 `Ctrl+B`
2. 规范化为思源 hotkey，例如 `⌘B`
3. 从 `window.siyuan.config.keymap` 反查命令
4. 如果命令正好属于稳定目录，则直接回退到稳定执行器
5. 如果是编辑器命令，则走 `editor-context runner`
6. 如果无法识别命令，再退回原始 `KeyboardEvent` 派发

这一步的关键不是“发按键”，而是“先理解它是什么命令”。

### B. `experimental-click-sequence`

推荐执行顺序：

1. 逐步解析配置的选择器
2. 每步等待元素出现
3. 每步检查可见性
4. 每步按重试策略执行
5. 步骤间按配置延迟
6. 任一步失败后按 `stopOnFailure` 决定终止或继续

它的定位应是：

- 自动化补位工具
- 复杂多步 UI 操作工具

而不是稳定命令系统的一部分。

## 7.3 快捷键映射的推荐算法

建议直接借鉴 `HaoCeans` 项目的思路，但以更清晰的模块边界实现：

1. `normalizeUserShortcut("Ctrl+Shift+P") -> "⇧⌘P"`
2. `findCommandByHotkey(window.siyuan.config.keymap, "⇧⌘P")`
3. 判断命令来源：
   - `general`
   - `editor.general`
   - `editor.insert`
   - `editor.heading`
   - `editor.list`
   - `editor.table`
   - `plugin`
4. 判断是否属于编辑器命令
5. 选择执行策略

## 7.4 编辑器命令的上下文规则

如果命令属于编辑器命令，推荐执行规则：

1. 获取活动编辑器
2. 保存选区
3. 恢复焦点
4. 恢复选区
5. 派发 `keydown`
6. 若失败，再尝试特定适配器

典型可补位场景：

- 加粗
- 斜体
- 标签
- 引用
- 行级代码
- 部分复制类命令

## 8. 点击序列设计

## 8.1 为什么要单独做点击序列

点击序列解决的是另一类问题：

- 某些操作不是一个命令，而是多步 UI 流程
- 某些功能没有稳定命令入口，但可以通过菜单和按钮组合触达
- 某些插件之间的联动天然只能通过 UI 自动化完成

因此它不应伪装成“命令”，而应被定义为独立动作类型。

## 8.2 推荐匹配策略

建议按如下顺序查找元素：

1. `id`
2. `data-id`
3. `data-menu-id`
4. `data-type`
5. `class`
6. SVG `use[href]` / `use[xlink:href]`
7. 文本精确匹配

推荐支持两种语法：

- 简单标识符：`barSettings`
- 显式文本：`text:复制块引用`
- 完整 CSS：`#barSettings`、`.b3-menu__item:nth-child(2)`

## 8.3 推荐执行规则

- 每步默认超时：`5000ms`
- 每步默认重试：`2`
- 每步默认重试间隔：`300ms`
- 每步默认成功后延迟：`200ms`

## 8.4 失败处理

实验点击序列失败时应：

- 给出错误提示
- 标记失败步骤序号
- 显示失败选择器
- 停止本次执行
- 不影响其他按钮

## 9. 原生工具栏实验适配策略

## 9.1 优先级判断

基于当前实现与两个参考项目的收益比，原生工具栏接管应排在：

1. `experimental-shortcut` 之后
2. `experimental-click-sequence` 之后

原因：

- 收益不如快捷键适配直观
- 风险高于快捷键适配
- DOM 结构改动影响面更大

## 9.2 如果未来要做

建议仅支持：

- 原生按钮显隐
- 少量顺序微调

仍不建议支持：

- 全量图标替换
- 全量原生按钮 metadata 抽取
- 原生工具栏完整接管

## 9.3 刷新时机建议

这块明确应借鉴 `HaoCeans` 的做法：

- 优先监听 EventBus
  - `loaded-protyle-dynamic`
  - `switch-protyle`
  - `loaded-protyle-static`
- MutationObserver 只做局部等待元素出现
- 不做全局常驻粗粒度扫描

## 10. 生命周期与渲染策略

## 10.1 稳定路径

建议继续保持：

1. `onload()`
   - 读取配置
   - 初始化命令目录
   - 注册插件命令
   - 订阅配置更新
2. `onLayoutReady()`
   - 创建 `SurfaceManager`
   - 渲染当前配置
3. `onunload()`
   - 销毁 surface
   - 销毁设置页
   - 清理订阅

## 10.2 实验路径

建议新增实验层生命周期：

1. 配置开启实验功能后再初始化
2. 注册局部 EventBus 监听
3. 用独立 adapter 持有所有观察器与定时器
4. 关闭实验功能或插件卸载时完整清理

## 11. 风险与应对

| 风险 | 等级 | 说明 | 应对 |
|---|---|---|---|
| `window.siyuan` 内部结构变化 | 高 | 影响快捷键反查、编辑器实例定位、实验命令执行 | 内部访问全部集中在实验模块，增加版本保护与失败降级 |
| 原生工具栏 DOM 结构变动 | 高 | 影响点击序列和原生工具栏适配 | 智能匹配 + 局部观察器 + 显式实验开关 |
| 编辑器焦点不正确 | 高 | 导致编辑器命令触发失败 | 增加 editor context runner、选区恢复与错误提示 |
| 快捷键与用户自定义映射冲突 | 中 | 同一快捷键在不同分类下映射不同命令 | 反查时记录命令来源并优先按 keymap 实际配置处理 |
| 移动端与桌面端行为不同 | 中 | 影响实验快捷键和点击序列 | 首版实验能力桌面端优先，移动端单独评估 |
| 配置演进导致旧配置失效 | 中 | 实验动作新增字段较多 | 引入版本迁移与字段兜底 |
| 自动化能力被误当成稳定能力 | 中 | 用户预期过高导致反馈噪音 | UI、README、设置页明确区分稳定与实验 |

## 12. 最终设计决策

当前阶段的正式设计决策如下：

1. 保持现有稳定核心架构不变。
2. 不把快捷键模拟并入默认命令执行链。
3. 参考 `HaoCeans` 的关键思想，增加“先从 keymap 识别命令，再执行”的实验快捷键层。
4. 参考 `HaoCeans` 的点击序列机制，增加独立的实验自动化动作类型。
5. 原生工具栏显隐/顺序控制继续作为低优先级实验能力，不提前进入主线。
6. EventBus 驱动的刷新时机优先级高于全局 MutationObserver。
