# 跨插件命令提供者协议设计

更新时间：`2026-04-11`

## 1. 背景

当前 [siyuan-power-buttons](/D:/MyCodingProjects/siyuan-power-buttons) 的“插件命令”仅支持本插件内部命令；[siyuan-doc-assist](/D:/MyCodingProjects/siyuan-doc-assist) 虽然已经拥有完整的文档处理动作体系与命令注册逻辑，但并未对外暴露稳定的集成接口，因此无法被快捷按钮直接发现、绑定和调用。

本设计的目标不是为 `siyuan-doc-assist` 做一次性的特判，而是定义一套通用的“跨插件命令提供者协议”：

- 任何插件都可以显式暴露一组允许外部绑定的公开命令；
- `power-buttons` 通过统一协议发现这些公开命令并绑定为按钮；
- 具体业务上下文、可用性判断和执行细节由提供者插件自己负责；
- 后续新增第三方插件时，不要求 `power-buttons` 为每个插件单独写耦合逻辑。

## 2. 目标与非目标

### 2.1 目标

- 支持 `power-buttons` 自动发现已安装插件中显式公开的命令提供者。
- 支持 `power-buttons` 绑定并执行 `siyuan-doc-assist` 暴露的文档处理命令。
- 允许提供者插件只公开“白名单”命令，而非默认公开全部内部动作。
- 将命令执行的上下文解析权保留在提供者插件内部。
- 为未来其他插件接入定义统一的接口、元数据、错误返回和兼容策略。

### 2.2 非目标

- 不让 `power-buttons` 直接读取其他插件的源码、构建产物或内部常量。
- 不要求 `power-buttons` 理解提供者插件的文档模型、编辑器模型或动作语义。
- 不尝试通过宿主全局命令面板反向推断某个命令是否适合外部绑定。
- 不把“所有已注册命令”自动视为“允许外部调用”。
- 不在第一阶段支持参数化命令、批量执行或跨插件回调链。

## 3. 核心设计原则

### 3.1 显式公开，默认私有

提供者插件必须显式声明哪些命令可以被 `power-buttons` 绑定。内部动作、实验性动作、强依赖特定 UI 状态的动作，默认不公开。

### 3.2 发现与执行解耦

`power-buttons` 只负责：

- 发现 provider；
- 展示 provider 公开的命令元数据；
- 记录按钮绑定关系；
- 在点击时调用 provider 的公开执行方法。

`power-buttons` 不负责：

- 解析当前文档 ID；
- 推断编辑器状态；
- 判断文档是否只读；
- 管理 provider 的确认弹窗与业务提示。

### 3.3 提供者自管上下文

用户已明确要求“外部命令自己决定如何解析上下文，`power-buttons` 只负责触发”。因此协议默认采用 `provider-managed context`：

- `power-buttons` 可以传递触发来源等轻量信息；
- 提供者插件必须自行解析当前活动编辑器、最近活跃文档、只读状态等运行上下文；
- 若上下文不足，错误提示由提供者插件或返回结果明确表达。

### 3.4 运行时集成，不依赖源码

发现路径统一基于 SiYuan 插件运行时的 `this.app?.plugins` 列表。`power-buttons` 只能与插件实例公开的协议对象交互，不得依赖 provider 的私有字段或内部模块路径。

### 3.5 协议优先于实现细节

为了允许多个插件接入，`power-buttons` 面向的是一份版本化协议，而不是面向 `doc-assist` 的具体类实现。

## 4. 总体架构

跨插件绑定能力拆分为三层。

### 4.1 Provider 层

任意希望被 `power-buttons` 绑定的插件，在自身插件实例上暴露一个版本化的 provider 工厂：

- `getPowerButtonsIntegration(): PowerButtonsCommandProvider | null`

该方法返回一份只包含公开能力的集成对象。

### 4.2 Discovery / Adapter 层

`power-buttons` 在运行时扫描 `this.app?.plugins`：

- 识别是否存在 `getPowerButtonsIntegration` 方法；
- 调用后验证返回对象是否满足协议；
- 将合法 provider 适配为内部统一结构，缓存到外部命令注册表中；
- 设置页和执行器均从该注册表读取 provider 信息。

### 4.3 Binding / Execution 层

用户在 `power-buttons` 中新建或编辑按钮时：

- 选择动作类型“外部插件命令”；
- 选择 provider 插件；
- 选择该 provider 公开命令。

按钮点击后：

- `power-buttons` 根据绑定信息定位 provider；
- 调用 provider 的 `invokeCommand()`；
- 根据 provider 返回结果决定是否追加 toast、是否刷新提示状态。

## 5. 协议定义

## 5.1 Provider 工厂入口

建议所有 provider 插件统一在插件实例上公开以下入口：

```ts
interface PowerButtonsIntegratablePlugin {
  getPowerButtonsIntegration?: () => PowerButtonsCommandProvider | null;
}
```

设计理由：

- 避免在插件实例上挂多个零散方法，降低命名污染；
- 允许提供者插件延迟构造 provider 对象；
- 可在后续版本通过工厂返回不同协议版本对象。

## 5.2 Provider 对象

```ts
interface PowerButtonsCommandProvider {
  protocol: 'power-buttons-command-provider';
  protocolVersion: 1;
  providerId: string;
  providerName: string;
  providerVersion?: string;
  listCommands: () => Promise<PowerButtonsPublicCommand[]> | PowerButtonsPublicCommand[];
  invokeCommand: (
    commandId: string,
    context: PowerButtonsInvokeContext,
  ) => Promise<PowerButtonsInvokeResult> | PowerButtonsInvokeResult;
}
```

字段语义：

- `protocol`：固定协议名，用于快速校验。
- `protocolVersion`：协议版本号；第一版固定为 `1`。
- `providerId`：稳定唯一标识，推荐等于插件 `name`，例如 `siyuan-doc-assist`。
- `providerName`：人类可读名称，用于设置页展示，例如 `文档助手 / Doc Assist`。
- `providerVersion`：可选，便于调试和兼容提示。
- `listCommands`：返回显式公开命令列表。
- `invokeCommand`：按命令 ID 执行公开命令。

## 5.3 公开命令定义

```ts
interface PowerButtonsPublicCommand {
  id: string;
  title: string;
  description?: string;
  category?: string;
  icon?: string;
  keywords?: string[];
  desktopOnly?: boolean;
  supportsMobile?: boolean;
}
```

字段约束：

- `id`：provider 内唯一命令 ID；必须稳定。
- `title`：设置页下拉和按钮绑定摘要使用的标题。
- `description`：帮助用户理解命令作用。
- `category`：用于分组展示，如 `导出`、`整理`、`编辑`。
- `icon`：可选，优先使用思源内置图标名或 SVG 文本；第一阶段可以只用于设置页列表展示，不强制映射到按钮图标。
- `keywords`：可用于未来命令搜索。
- `desktopOnly` / `supportsMobile`：可用性提示，仅用于展示；最终是否可执行仍由 provider 决定。

注意：

- 此结构只描述“公开命令元数据”，不暴露内部处理器引用。
- 不在第一版加入参数描述或动态表单，避免过早泛化。

## 5.4 调用上下文

```ts
interface PowerButtonsInvokeContext {
  trigger: 'button-click';
  sourcePlugin: 'siyuan-power-buttons';
  sourcePluginVersion?: string;
  surface?: string;
  buttonId?: string;
}
```

上下文规则：

- `power-buttons` 只传递触发来源、表面位置、按钮 ID 这类轻量信息；
- 不传 `docId`、`blockId`、`protyle` 等 provider 业务上下文；
- provider 插件必须自行完成上下文解析。

## 5.5 执行结果

```ts
interface PowerButtonsInvokeResult {
  ok: boolean;
  message?: string;
  alreadyNotified?: boolean;
  errorCode?:
    | 'command-not-found'
    | 'provider-unavailable'
    | 'context-unavailable'
    | 'not-supported'
    | 'execution-failed';
}
```

约束：

- `ok: true` 表示 provider 已接管并成功执行，或成功启动异步流程。
- `message` 用于给 `power-buttons` 追加统一提示。
- `alreadyNotified: true` 表示 provider 已自行调用 `showMessage` 或弹窗，`power-buttons` 不应重复 toast。
- `errorCode` 只用于日志和调试，不作为本地化消息主来源。

## 6. `power-buttons` 侧设计

## 6.1 动作类型扩展

新增动作类型：

- `external-plugin-command`

这样可以与现有的本插件 `plugin-command` 明确区分，避免语义混淆。

`ACTION_TYPES` 将变为：

- `builtin-global-command`
- `plugin-command`
- `external-plugin-command`
- `open-url`
- `experimental-shortcut`
- `experimental-click-sequence`

## 6.2 配置模型

建议第一版继续复用已有 `actionId` 字段，采用稳定复合 ID 编码：

```ts
actionType: 'external-plugin-command'
actionId: 'siyuan-doc-assist:insert-doc-summary'
```

原因：

- 可减少一次结构性配置迁移；
- 保持导入导出逻辑简单；
- 与当前 `PowerButtonItem` 结构兼容；
- 缺失 provider 时仍可保留原始绑定信息。

同时增加纯函数工具：

- `formatExternalCommandActionId(providerId, commandId)`
- `parseExternalCommandActionId(actionId)`
- `isExternalCommandActionId(actionId)`

## 6.3 外部命令注册表

`power-buttons` 新增 `ExternalCommandRegistry`，负责：

- 扫描 `this.app?.plugins`；
- 发现 provider 工厂；
- 校验协议对象；
- 拉取并缓存 provider 命令列表；
- 提供按 `providerId` / `providerId + commandId` 查询；
- 在执行失败或设置页打开时支持主动刷新。

建议职责：

- **发现逻辑**：独立于设置页和执行器；
- **缓存逻辑**：避免设置页每次渲染重复扫描；
- **兼容逻辑**：屏蔽非法 provider 实现对上层的影响。

## 6.4 设置页交互

在按钮编辑区新增以下交互：

当 `actionType === 'external-plugin-command'` 时，显示：

1. provider 下拉框
2. command 下拉框
3. 命令描述 / 可用性提示区域
4. 刷新外部插件命令列表按钮

设置页行为：

- 若未发现任何 provider，显示“未检测到支持外部绑定的插件”。
- 若当前配置绑定的 provider 未安装，保留原值并显示“插件未安装或未暴露协议”。
- 若 provider 存在但 command 已下线，保留原值并显示“命令已不存在或不再公开”。
- 若 provider 返回了 `description`、`category`，在选择器下方展示摘要。

## 6.5 执行逻辑

点击按钮时，`CommandExecutor` 对 `external-plugin-command` 分支执行以下流程：

1. 解析 `actionId` 获得 `providerId` 与 `commandId`
2. 从 `ExternalCommandRegistry` 查找 provider
3. provider 缺失时：
   - 尝试刷新一次注册表
   - 仍缺失则提示“未检测到对应插件或版本不兼容”
4. 调用 `provider.invokeCommand(commandId, context)`
5. 根据返回值：
   - `ok && !alreadyNotified && message`：显示成功提示
   - `!ok && !alreadyNotified && message`：显示错误提示
   - `alreadyNotified`：不重复提示

## 6.6 导入导出与迁移

- 导出 JSON 时直接保留 `actionType` 和复合 `actionId`
- 导入时允许导入“当前未安装 provider”的配置
- 配置校验层只验证：
  - `actionType` 合法
  - `actionId` 格式可解析
- 不在导入阶段强制要求 provider 实时存在

这样可以保证：

- 跨设备同步配置可行
- 用户先导入配置，再安装对应插件也能恢复工作

## 7. `siyuan-doc-assist` 侧设计

## 7.1 Provider 暴露位置

在插件生命周期类 [plugin-lifecycle.ts](/D:/MyCodingProjects/siyuan-doc-assist/src/plugin/plugin-lifecycle.ts) 上新增：

```ts
public getPowerButtonsIntegration(): PowerButtonsCommandProvider | null
```

该方法返回一个稳定 provider 对象，内部闭包绑定 `actionRunner`、插件版本、公开命令映射。

## 7.2 公开命令白名单

新增专用模块维护“对外公开命令白名单”，例如：

- `src/plugin/power-buttons-provider.ts`
- `src/plugin/power-buttons-provider-types.ts`

模块职责：

- 定义公开协议类型；
- 声明 `PUBLIC_DOC_ASSIST_COMMANDS`；
- 建立 `public command id -> ActionKey` 映射；
- 构造 `listCommands()` 返回值；
- 处理 `invokeCommand()` 路由。

白名单必须显式列出，而不是从 `ACTIONS` 自动推导全部命令。

## 7.3 公开命令选择原则

第一阶段仅公开适合快捷按钮直接触发、语义清晰、对上下文要求可由 provider 自行解决的动作。建议优先从以下类别中选择：

- 导出类：
  - `export-current`
  - `export-child-docs-zip`
  - `export-child-key-info-zip`
- 插入/整理类：
  - `insert-backlinks`
  - `insert-child-docs`
  - `create-open-docs-summary`
- 清理/编辑类：
  - `clean-ai-output`
  - `trim-trailing-whitespace`
  - `remove-extra-blank-lines`
  - `toggle-links-refs`
  - `insert-doc-summary`

第一阶段不建议公开：

- 高度依赖当前选区的命令；
- 强依赖文档标题菜单上下文的命令；
- 需要复杂二次确认且用户难以预期结果的命令；
- 实验性或 alpha 隐藏功能；
- 仅适用于某些前端或某些插件联动场景的命令。

## 7.4 命令 ID 设计

建议 `doc-assist` 对外命令 ID 不直接等于 `ActionKey`，而是保留一层“公开命令 ID”映射。推荐规则：

```ts
public id: 'doc-assist.insert-doc-summary'
action key: 'insert-doc-summary'
```

这样可以：

- 避免未来内部 `ActionKey` 重命名直接破坏外部协议；
- 允许一个公开命令对接内部不同实现；
- 为未来 provider 拆分或命令重组保留空间。

若需要控制第一阶段复杂度，也可以先让公开 ID 与 `ActionKey` 相同，但必须在文档中声明“公开 ID 从协议视角看视为稳定 API，不应轻易改名”。

推荐做法：

- 第一阶段直接复用 `ActionKey` 作为 `commandId`
- 但单独维护白名单映射，后续若内部重构再引入显式协议 ID

## 7.5 执行语义

`invokeCommand()` 执行职责：

1. 校验 `commandId` 是否存在于公开白名单
2. 检查当前前端环境是否允许执行
3. 由 `doc-assist` 自行解析当前文档上下文
4. 调用内部 `actionRunner.runAction()`
5. 将执行结果转换为协议结果

返回约束：

- 命令不存在：`ok: false`, `errorCode: 'command-not-found'`
- 当前无可用文档上下文：`ok: false`, `errorCode: 'context-unavailable'`
- provider 已自行提示用户：`alreadyNotified: true`
- 执行成功但无需重复提示：`ok: true`, `alreadyNotified: true`

## 7.6 与现有命令注册共存

`doc-assist` 现有命令面板注册逻辑保持不变：

- 命令面板 / 快捷键入口继续使用自身 `registerPluginCommands()`
- 新 provider 层只增加外部集成能力
- 不要求将外部 provider 与宿主命令注册合并成一套机制

这可以避免为支持 `power-buttons` 而重写 `doc-assist` 原有命令体系。

## 8. 通用兼容与失败策略

## 8.1 Provider 缺失

场景：

- 用户配置了某个 provider 命令，但目标插件未安装
- 目标插件已禁用
- 目标插件版本过老，未暴露协议

处理：

- 设置页保留原绑定值并显示缺失警告
- 运行时点击按钮提示“未检测到对应插件或版本不兼容”
- 不自动删除配置

## 8.2 Command 缺失

场景：

- provider 仍存在，但某命令不再公开

处理：

- 设置页显示“命令已不存在或不再公开”
- 配置保留，等待用户手动改绑
- 执行时报 `command-not-found`

## 8.3 协议版本不兼容

场景：

- provider 返回 `protocolVersion !== 1`

处理：

- `power-buttons` 将该 provider 视为不可用；
- 设置页可显示“协议版本不兼容”；
- 日志记录 `providerId`、`providerVersion`、`protocolVersion`。

## 8.4 Provider 异常

场景：

- `listCommands()` 抛异常
- `invokeCommand()` 抛异常

处理：

- `power-buttons` 捕获异常并转成统一错误提示；
- 不允许异常冒泡破坏按钮渲染或设置页；
- 开发模式下可打印详细错误对象。

## 9. 安全与边界

本协议默认信任“用户已安装的本地插件”，但仍需限制边界：

- `power-buttons` 不做任意反射调用，只调用标准 provider 接口；
- provider 只公开显式白名单命令；
- `power-buttons` 不把任意宿主内部对象直接透传给 provider；
- provider 若要做危险操作，仍应沿用自身确认逻辑与只读保护。

## 10. 测试设计

## 10.1 `power-buttons` 测试

新增测试覆盖：

- 发现合法 provider 并读取命令列表
- 忽略非法 provider（缺字段、错协议版本、抛异常）
- `external-plugin-command` 的 `actionId` 解析与格式化
- 设置页在 provider 存在 / 缺失 / 命令缺失场景下的展示
- 点击按钮后成功调用 `invokeCommand`
- `alreadyNotified` 分支不重复 toast
- provider 缺失时刷新一次再报错
- 导入导出包含 `external-plugin-command` 的配置

## 10.2 `siyuan-doc-assist` 测试

新增测试覆盖：

- `getPowerButtonsIntegration()` 返回合法 provider 对象
- `listCommands()` 只返回白名单命令
- 被隐藏/不公开动作不会出现在 provider 列表
- `invokeCommand()` 能把公开命令正确路由到 `ActionRunner`
- 未知命令返回 `command-not-found`
- 上下文不足时返回 `context-unavailable`
- provider 执行异常时返回 `execution-failed`

## 10.3 联合集成测试

在 `power-buttons` 侧增加 provider mock，模拟：

- `siyuan-doc-assist` 已安装并公开 3 个命令
- 版本不兼容
- provider 执行时报错

无需在第一阶段建立真正跨仓库集成测试，只要协议 mock 与双方单测契约一致即可。

## 12. 分阶段落地计划

### Phase 1：协议落地与 `doc-assist` 首次接入

- `power-buttons` 定义 provider 协议与注册表
- `power-buttons` 设置页支持 `external-plugin-command`
- `doc-assist` 暴露 provider 并公开第一批白名单命令
- 完成双方单测和 `power-buttons` mock 集成测试

交付标准：

- 用户在同时安装两个插件时，可在 `power-buttons` 设置页看到 `文档助手 / Doc Assist`
- 可选择至少一条公开命令并成功通过按钮触发

### Phase 2：展示增强

- provider 命令列表支持分类展示和搜索
- 设置页展示更完整命令描述、桌面端限制提示
- 缺失 provider / 命令的状态提示更明确

### Phase 3：生态扩展

- 抽离协议说明，供其他插件实现
- 新 provider 插件可在不改 `power-buttons` 业务代码的前提下接入

## 13. 决策总结

本设计采用：

- **方案 C：provider 插件主动暴露协议，`power-buttons` 统一发现与调用**

最终原因：

- 边界最清晰；
- 与 `doc-assist` 当前通过 `this.app?.plugins` 发现其他插件的模式一致；
- 能通过显式白名单控制公开命令；
- 允许 `power-buttons` 保持通用，而不是为单一插件耦合实现；
- 能为未来第三方插件提供可复制的接入路径。

## 14. 后续执行建议

实现时建议先做协议与 mock，再接入 `doc-assist` 真 provider，避免边写边改两边接口。

推荐实现顺序：

1. 在 `power-buttons` 中先定义协议类型、注册表和设置页状态结构；
2. 用测试内 provider mock 跑通发现、绑定、执行、错误提示；
3. 在 `doc-assist` 中按白名单实现真实 provider；
4. 回到 `power-buttons` 做联调与兼容验证；
5. 最后补文档与示例配置。
