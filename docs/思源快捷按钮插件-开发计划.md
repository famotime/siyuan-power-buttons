# 思源快捷按钮插件开发计划

- 文档日期：2026-04-05
- 开发模板：`plugin-sample-vite-vue`
- 目标版本：V1 桌面端稳定版
- 适配基线：SiYuan `v3.5.7`

## 1. 目标定义

V1 的交付目标不是“控制思源全部原生工具栏”，而是：

- 建立一套插件自有的快捷按钮系统
- 支持按钮渲染到顶栏、左/右/底 Dock、底栏状态区
- 支持可视化配置、图标选择、命令绑定、拖拽排序、持久化
- 支持调用一组稳定可维护的内置全局命令和插件命令

## 2. 非目标

V1 不纳入以下承诺：

- 稳定控制全部原生顶栏按钮
- 稳定控制全部原生 Dock 按钮
- 稳定替换原生图标
- 覆盖命令面板全部上下文命令
- 完整移动端支持

```text

  App.vue

    icons/
```

建议职责分配：

- `src/index.ts`
  - 插件类

- `src/core/config`
  - 配置 schema
  - 默认值
  - 存储与迁移
- `src/core/commands`
  - 命令目录
  - 执行器

- `src/core/surfaces`
  - 顶栏渲染
  - 状态栏渲染
  - Dock 渲染
- `src/ui/settings`
  - 配置页面
  - 按钮编辑器

  - 命令选择器

## 4. 里程碑计划

## 4.1 里程碑 M0：模板收敛与基线清理

目标：

- 把模板从示例工程改成可开发基线

任务：

- 替换模板插件名、`plugin.json`、README 元信息
- 清理 `App.vue` 里的示例交互代码
- 保留 Vue 挂载方式，但改为服务于设置界面
- 确认 `onLayoutReady()` 为按钮渲染入口
- 建立统一日志开关和错误提示工具

输出：

- 可启动的空白插件骨架
- 明确的目录结构

验收：

- `pnpm dev` 可正常构建
- 插件可在思源中启用/停用
- 无示例脏逻辑遗留

## 4.2 里程碑 M1：配置模型与持久化

目标：

- 定义 V1 的稳定配置模型

任务：

- 定义 `PowerButtonsConfig`、`PowerButtonItem`
- 建立默认配置生成器
- 建立配置校验器
- 建立配置迁移器
- 封装 `loadData/saveData/removeData`
- 建立内存态 store

建议实现点：

- 配置持久化文件建议使用单一 JSON 文件
- 首版就加入 `version`
- 对非法图标、非法命令、未知 surface 做清洗

输出：

- `ConfigStore`
- 默认配置
- 配置读写与迁移逻辑

验收：

- 修改配置后禁用/启用插件仍能恢复
- 非法配置不会导致渲染崩溃

## 4.3 里程碑 M2：稳定命令目录与执行器

目标：

- 建立“受支持命令目录”

任务：

- 从官方 `global.ts` 整理出稳定命令清单
- 定义命令元数据：
  - `id`
  - `title`
  - `category`
  - `surfaceSuggestion`
  - `requiresContext`
  - `stability`
- 封装统一执行器
- 增加插件自定义命令注册层
- 约定点击按钮与命令面板统一复用执行逻辑

建议首版命令分类：

- 导航类
- 面板类
- 搜索类
- 标签页类
- 系统类
- 插件类

输出：

- `BuiltinCommandCatalog`
- `CommandExecutor`
- `PluginCommandRegistry`

验收：

- 选中的稳定命令可以从按钮点击正常执行
- 插件命令和按钮动作走同一套执行链

## 4.4 里程碑 M3：区域渲染引擎

目标：

- 让按钮能按配置渲染到目标区域

任务：

- 实现顶栏按钮渲染器
- 实现状态栏按钮渲染器
- 实现 Dock 按钮渲染器
- 维护按钮实例引用表
- 实现幂等刷新：
  - 全量重建
  - 局部更新
- 处理禁用/卸载时销毁

设计重点：

- 顶栏和状态栏采用轻量点击按钮
- Dock 入口可承载更复杂面板
- 需要统一的 `render(config)` 与 `destroy()` 接口

输出：

- `TopBarSurfaceRenderer`
- `StatusBarSurfaceRenderer`
- `DockSurfaceRenderer`
- `SurfaceManager`

验收：

- 按钮可出现在正确区域
- 顺序正确
- 显隐切换后 UI 正确刷新
- 插件卸载后不残留 DOM

## 4.7 里程碑 M6：实验特性适配层

- 在不影响稳定版的前提下，为原生按钮控制预留扩展

任务：

- 增加实验开关

- 识别原生 Dock 按钮类型
- 封装原生按钮显隐控制
- 封装失败降级逻辑

- 原生按钮显示/隐藏

不建议在这一阶段做：

输出：

- `CompatibilityAdapter`

验收：

- 关闭实验开关后不会影响稳定功能
- 新版本失效时能自动降级而不是报错

## 5. 推荐开发顺序

建议按以下顺序推进，不要一开始就碰实验特性：

1. M0 模板清理
2. M1 配置模型
3. M2 命令目录
4. M3 区域渲染
5. M4 可视化设置
6. M5 导入导出
7. M7 测试发布
8. M6 实验增强

原因：

- 稳定核心功能先闭环，能尽快拿到可用版本
- 实验特性放后面，不阻塞主线交付

## 6. 关键技术决策

## 6.1 生命周期决策

- `onload()` 负责加载配置和准备服务
- `onLayoutReady()` 负责渲染顶栏、状态栏、Dock

## 6.2 命令策略决策

- V1 只正式支持 `globalCommand` 目录和插件命令
- 其他命令只作为后续增强

## 6.3 区域策略决策

- 顶栏只承载少量高频入口
- 大量快捷入口优先放 Dock
- 底栏只承载轻量动作

## 6.4 兼容策略决策

- 所有实验特性默认关闭
- 内部 DOM 依赖全部集中在 `compatibility` 层
- 遇到未知版本直接降级

## 7. 测试清单

建议至少执行以下手工用例：

1. 新建 3 个按钮，分别放到顶栏、左 Dock、底栏，检查位置是否正确
2. 修改图标后立即查看预览与实际渲染是否一致
3. 修改命令绑定后点击按钮，检查命令是否执行
4. 隐藏按钮后确认界面立即更新
5. 调整排序后确认顺序变化正确
6. 导出配置后删除按钮，再导入，确认恢复成功
7. 禁用插件、重新启用，确认配置恢复
8. 卸载插件后确认按钮不残留
9. 打开实验开关后启用原生按钮显隐控制，再关闭，确认可恢复

## 8. 版本规划

建议分三版：

### V0.1

- 完成稳定核心最小闭环
- 支持少量按钮和稳定命令

### V0.2

- 完成可视化配置
- 完成图标选择与导入导出

### V1.0

- 完成稳定版发布
- 形成正式文档与发布包

### V1.x

- 按实验开关逐步加入原生按钮适配

## 9. 最终建议

开发时要坚持一条原则：

先把“插件自有按钮系统”做成一个稳定产品，再决定要不要进一步“接管思源原生工具栏”。

按这个计划推进，项目风险可控，且能较快交付一个真正有用的首版。
