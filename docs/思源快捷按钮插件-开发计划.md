# 思源快捷按钮插件开发计划

- 文档日期：2026-04-05
- 当前版本状态：V1 稳定版已完成
- 下一阶段目标：V1.1 实验快捷键层，V1.2 实验点击序列层
- 适配基线：SiYuan `v3.5.7`

## 1. 当前阶段判断

项目主线已经不是“从零做首版”，而是：

1. 保持现有稳定核心可持续维护
2. 在稳定架构上追加实验能力
3. 明确把实验能力收口到独立模块、独立配置、独立测试

因此后续计划不再按最初的 M0-M7 从头描述，而按“已完成基线 + 下一阶段实施计划”组织。

## 2. 已完成的基线

当前仓库已完成：

- 配置模型与持久化
- 稳定命令目录
- 统一执行器
- 顶栏、状态栏、Dock 渲染
- 设置界面
- JSON 导入导出
- 单元测试基础覆盖
- `package.zip` 生成

当前代码中可直接复用的稳定模块：

- `src/core/config`
- `src/core/commands/catalog.ts`
- `src/core/commands/executor.ts`
- `src/core/commands/builtin-dom.ts`
- `src/core/surfaces`
- `src/shared/types.ts`
- `tests/command-executor.test.ts`
- `tests/builtin-dom.test.ts`

## 3. 后续开发总目标

后续开发要达成的不是“扩大稳定承诺范围”，而是：

- 让实验能力在不污染稳定链路的前提下可用
- 让用户明确知道什么是稳定能力，什么是实验能力
- 让后续维护工作有清晰边界

## 4. 下一阶段版本规划

建议后续拆成 4 个连续阶段：

### V1.1

- 新增 `experimental-shortcut`
- 不做点击序列
- 不做原生工具栏接管

### V1.2

- 新增 `experimental-click-sequence`
- UI 中加入高级实验配置区

### V1.3

- 视实际收益，考虑少量原生工具栏显隐适配
- 仍不做全量原生工具栏接管

### V1.4

- 真实桌面端回归
- 文档、示例、发布说明完善

## 5. 分阶段实施计划

目标：

- 把当前代码、README、设置页文案与最新设计决策对齐

任务：

- 更新 `docs/思源快捷按钮插件-设计方案.md`

输出：

- 当前产品边界表达统一

验收：

- 文档中已明确稳定层与实验层的边界
- 文档中已明确借鉴点和不采纳点

## 5.2 阶段 B：`experimental-shortcut` 基础设施

目标：

- 在现有执行器旁边增加“实验快捷键适配器”，但不影响稳定链

建议新增文件：

- `src/core/commands/experimental-shortcut.ts`
- `src/core/commands/shortcut-keymap.ts`
- `src/core/commands/editor-context.ts`
- `tests/experimental-shortcut.test.ts`
- `tests/shortcut-keymap.test.ts`

任务拆解：

1. 扩展类型系统
   - 更新 `src/shared/types.ts`
   - 新增 `experimental-shortcut`
   - 新增实验配置字段

2. 增加配置清洗与迁移
   - 更新 `src/core/config/sanitize.ts`
   - 更新 `src/core/config/defaults.ts`
   - 更新 `src/core/config/store.ts`
   - 决定是否将配置版本提升到 `2`

3. 实现快捷键规范化
   - 用户输入 `Ctrl+B`
   - 转为思源 hotkey `⌘B`
   - 统一修饰键顺序

4. 实现 keymap 反查
   - 查找 `window.siyuan.config.keymap.general`
   - 查找 `editor.general`
   - 查找 `editor.insert`
   - 查找 `editor.heading`
   - 查找 `editor.list`
   - 查找 `editor.table`
   - 查找 `plugin`

5. 实现编辑器上下文执行
   - 获取活动编辑器
   - 保存选区
   - 恢复焦点
   - 恢复选区
   - 向编辑区派发 `keydown`

6. 接入统一执行器
   - 更新 `src/core/commands/executor.ts`
   - 新增实验分支
   - 保持稳定分支不变

7. 增加错误提示
   - 快捷键解析失败
   - keymap 未命中
   - 找不到活动编辑器
   - 执行失败但已降级

设计要求：

- 如果反查出的命令已经是稳定目录里的命令，必须优先回到稳定执行器，而不是继续走实验事件模拟
- 只有在稳定执行器覆盖不到时，才继续走实验快捷键模拟

输出：

- `experimental-shortcut` 基础实现
- 类型系统与配置系统完成扩展

验收：

- 不开启实验开关时，现有所有测试仍通过
- 开启实验开关后，能执行至少一组编辑器快捷键
- 关闭实验开关后，行为完全回退到现状

## 5.3 阶段 C：设置界面接入实验快捷键

目标：

- 让用户能配置实验快捷键，但默认看不到或默认不启用

建议修改文件：

- `src/App.vue`
- `src/shared/constants.ts`
- `src/core/config/import-export.ts`
- `tests/import-export.test.ts`
- `tests/config-store.test.ts`

任务拆解：

1. 扩展动作类型下拉
   - 增加 `experimental-shortcut`
   - 在文案中标记“实验”

2. 增加快捷键配置字段
   - 文本输入框或录制输入框
   - `sendEscapeBefore`
   - `dispatchTarget`

3. 增加实验开关区
   - `shortcutAdapter`
   - 显示风险说明

4. 更新导入导出
   - 支持新字段
   - 非法字段回退

5. 更新预览与提示
   - 预览中可看到动作类型
   - 编辑器中可看到实验标签

验收：

- 不开启实验模式时，用户无法误用实验动作
- 导入旧配置不会崩
- 导出新配置后再导入可以完整恢复

## 5.4 阶段 D：`experimental-click-sequence`

目标：

- 增加独立的点击序列自动化动作类型

建议新增文件：

- `src/core/commands/click-sequence.ts`
- `src/core/commands/dom-query.ts`
- `tests/click-sequence.test.ts`
- `tests/dom-query.test.ts`

任务拆解：

1. 扩展动作类型
   - 新增 `experimental-click-sequence`

2. 实现元素匹配器
   - `id`
   - `data-id`
   - `data-menu-id`
   - `data-type`
   - `class`
   - SVG `use[href]`
   - `text:xxx`
   - 完整 CSS 选择器

3. 实现等待器
   - 元素已存在直接返回
   - 局部 `MutationObserver` 等待
   - 超时退出

4. 实现重试器
   - 默认重试 `2`
   - 默认延迟 `300ms`

5. 实现点击执行器
   - 优先 `element.click()`
   - 失败时派发 `MouseEvent("click")`

6. 接入统一执行器
   - 只在 `experimental-click-sequence` 分支调用

7. 接入设置页
   - 多行步骤编辑
   - 每步参数配置
   - 文案说明

设计要求：

- 点击序列必须被视为“自动化动作”，不能伪装成内置命令
- 默认仅面向桌面端
- 任何失败都不能影响其他按钮

输出：

- 可配置点击序列执行器

验收：

- 可正确执行简单两步序列
- 步骤失败时能报告是哪一步失败
- 不会导致界面卡死或残留观察器

## 5.5 阶段 E：原生工具栏低优先级适配

目标：

- 仅在确有必要时，补少量原生按钮显隐或顺序微调能力

建议新增文件：

- `src/core/compatibility/native-toolbar.ts`
- `src/core/compatibility/version-guard.ts`
- `tests/native-toolbar-adapter.test.ts`

任务拆解：

1. 建立版本保护
   - 允许版本白名单
   - 不匹配时直接退出

2. 识别原生工具栏节点
   - 顶栏
   - 状态栏
   - 少量原生按钮

3. 只做显隐与有限顺序调整
   - 不做图标替换
   - 不做完整接管

4. 严格清理副作用
   - 关闭实验开关
   - 卸载插件
   - 配置回滚

设计要求：

- 这是低优先级阶段
- 如果阶段 B、C、D 已满足用户主要诉求，可以无限期延后

验收：

- 实验开关关闭后界面恢复
- 版本不匹配时静默降级

## 6. 推荐的文件级改造顺序

建议按以下文件顺序推进，避免一次改动面过大：

1. `src/shared/types.ts`
2. `src/core/config/defaults.ts`
3. `src/core/config/sanitize.ts`
4. `src/core/config/store.ts`
5. `src/core/commands/executor.ts`
6. 新增 `src/core/commands/shortcut-keymap.ts`
7. 新增 `src/core/commands/editor-context.ts`
8. 新增 `src/core/commands/experimental-shortcut.ts`
9. `src/App.vue`
10. `src/core/config/import-export.ts`
11. 补测试
12. 再进入点击序列阶段

原因：

- 先打通类型与配置
- 再扩展执行器
- 最后接 UI
- 测试要跟着每个小阶段走

## 8. UI 与文案计划

实验能力必须在 UI 上明显区别于稳定能力。

建议做法：

- 在动作类型中显示：
  - `实验：快捷键适配`
  - `实验：点击序列`
- 在右侧属性编辑区加入警告文案：
  - 依赖当前焦点
  - 依赖当前界面结构
  - 不保证跨版本稳定
- 在实验开关旁边加入描述：
  - “默认关闭”
  - “可能随思源版本变化失效”

## 10. 决策清单

当前后续开发的关键决策如下：

1. 不重写现有稳定架构。
2. 不把快捷键重放变成默认执行路径。
3. 先做 `experimental-shortcut`，再做 `experimental-click-sequence`。
4. 原生工具栏适配延后。
5. EventBus 优先，Observer 补位。
6. 每新增一种实验能力，必须同步补测试和配置迁移。
