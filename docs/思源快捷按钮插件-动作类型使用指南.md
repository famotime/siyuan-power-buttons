# 随心按插件动作类型使用指南

- 文档日期：2026-04-15
- 适用版本：当前仓库实现基线，插件 `minAppVersion = 3.5.7`

本文面向插件使用者，重点说明“按钮点击后到底会执行什么”。其中会特别展开两个实验动作类型：

- `实验：快捷键适配`
- `实验：点击序列`

如果你只想快速上手，先看“快速开始”和后面的示例；如果你要排查为什么某个按钮不生效，直接看“常见问题与排查”。

## 1. 动作类型总览

当前代码模型支持以下动作类型：

| 动作类型 | 用途 | 稳定性 | 适合场景 |
| --- | --- | --- | --- |
| `内置命令` | 执行思源内置全局命令 | 稳定 | 全局搜索、设置、窗口级操作 |
| `插件命令` | 执行本插件或其他插件暴露的命令 | 稳定 | 打开本插件设置、调用外部插件能力 |
| `实验：快捷键适配` | 把一个按钮映射为“按一次快捷键”或“按键语义回放” | 实验 | 编辑器格式化、快捷键已有但无稳定 API 的动作 |
| `实验：点击序列` | 按步骤查找界面元素并依次点击 | 实验 | 菜单链路、弹出菜单后二次点击、原生入口自动化 |
| `打开链接` | 打开 URL | 模型已支持 | 当前设置页未暴露该选项 |

注意：

- 当前设置页的动作类型下拉框里，已经去掉了 `打开链接`，所以你在界面里通常只会看到前四项。
- 如果能用 `内置命令` 或 `插件命令` 解决，不建议优先使用实验动作。

## 2. 快速开始

### 2.1 新建一个按钮

1. 打开插件设置。
2. 在左侧按钮列表中新建或选中一个按钮。
3. 在右侧“按钮设置”中填写：
   - 按钮名称
   - 提示文字
   - 显示位置
   - 动作类型
4. 根据动作类型继续填写对应配置。

### 2.2 选型建议

- 想执行思源已公开的全局命令：选 `内置命令`
- 想调用某个插件提供的能力：选 `插件命令`
- 想把“Ctrl+B / Ctrl+Shift+P”之类的快捷键挂到按钮上：选 `实验：快捷键适配`
- 想模拟“先点 A，再点 B，再点 C”：选 `实验：点击序列`

## 3. 各动作类型说明

### 3.1 内置命令

这是最推荐的动作类型。

特点：

- 优先走思源稳定命令执行链
- 失败时仅对少量内置命令走 DOM 兜底
- 对界面结构依赖最小

适用场景：

- 全局搜索
- 打开设置
- 其他已在插件命令目录中列出的稳定命令

建议：

- 如果目标动作能在这里找到，就不要改用实验动作。

### 3.2 插件命令

用于执行：

- 本插件内部命令
- 其他插件通过 provider 协议公开的命令

设置方式：

1. 动作类型选择 `插件命令`
2. 先选“插件名称”
3. 再选“插件命令”
4. 如有需要，点“刷新插件命令”

适用场景：

- 打开本插件设置
- 调用其他插件的整理、插入、摘要、转换等命令

建议：

- 只要目标插件愿意公开命令，这通常比点击序列稳定得多。

### 3.3 打开链接

代码执行器仍支持 `open-url`，但当前设置页下拉框已隐藏这个动作类型。

这意味着：

- 运行时模型支持它
- 当前图形设置界面不提供直接选择
- 如确实需要，只能通过配置文件导入等方式使用

## 4. 实验动作的启用条件

`实验：快捷键适配` 和 `实验：点击序列` 都会在运行时经过保护判断。

只有同时满足以下条件时，实验动作才会真正执行：

1. 对应实验开关已启用
2. 当前前端是桌面端
3. 思源版本不低于 `3.5.7`

如果条件不满足，点击按钮时会直接提示错误，而不是静默失败。

### 4.1 关于实验开关

当前实现里，设置页会显示实验能力是否启用，但没有单独提供明显的开关表单。若你看到以下提示：

- `实验快捷键适配当前未启用`
- `实验点击序列当前未启用`

可通过以下方式恢复：

1. 导出插件配置 JSON
2. 将 `experimental.shortcutAdapter` 或 `experimental.clickSequenceAdapter` 改为 `true`
3. 再导入配置

参考片段：

```json
{
  "experimental": {
    "nativeToolbarControl": true,
    "internalCommandAdapter": true,
    "shortcutAdapter": true,
    "clickSequenceAdapter": true
  }
}
```

## 5. 重点一：实验“快捷键适配”

### 5.1 这是什么

它不是简单的“总是往页面发一个键盘事件”。

当前实现会按以下顺序处理：

1. 先把你录入的快捷键规范化为思源 keymap 格式
2. 尝试去 `window.siyuan.config.keymap` 里反查这个快捷键对应的命令
3. 如果反查到的是内置全局命令，优先走稳定命令执行
4. 如果反查到的是插件命令，优先走插件命令执行
5. 如果反查不到稳定命令，再根据派发目标把 `keydown` 事件发给编辑器、`window` 或 `body`

所以它比“纯键盘模拟”更稳一点，但本质上仍然是实验能力。

### 5.2 如何配置

在右侧动作设置中，选择 `实验：快捷键适配` 后，可以配置：

| 字段 | 说明 |
| --- | --- |
| `快捷键` | 焦点放到输入框后，直接按组合键录入 |
| `派发目标` | 决定按键事件优先发给哪里 |
| `先发 Escape` | 在真正派发快捷键前，先向窗口发送一次 `Escape` |
| `允许直接发到 window` | 当派发目标为自动时，允许退回到 `window` |

### 5.3 快捷键录入规则

录入框是“捕获输入”而不是普通文本框。

行为规则：

- 聚焦输入框后，直接按组合键
- `Backspace` 或 `Delete` 可以清空当前快捷键
- 无修饰键时，只有 `F1` 到 `F12` 这类功能键允许单独录入
- 常见可识别主键包括：
  - 字母，如 `B`
  - 数字，如 `5`
  - 功能键，如 `F2`
  - 特殊键，如 `Escape`、`Enter`、`Tab`、`ArrowUp`

示例：

- 按下 `Ctrl+B`，会保存为 `Ctrl+B`
- 按下 `Ctrl+Shift+P`，会保存为 `Ctrl+Shift+P`
- 按下 `Alt+5`，会保存为 `Alt+5`

### 5.4 快捷键归一化

运行时会把人类可读写法转换为思源内部 hotkey 形式，例如：

| 录入值 | 归一化结果 |
| --- | --- |
| `Ctrl+B` | `⌘B` |
| `Ctrl+Shift+P` | `⇧⌘P` |
| `Alt+5` | `⌥5` |

注意：

- `Ctrl`、`Cmd`、`Command`、`Meta` 在归一化时会被视为同一类主修饰键
- 这也意味着冲突检测会按归一化后的结果来判断

例如：

- 一个按钮用了 `Ctrl+B`
- 另一个按钮再设置成 `Cmd+B`

在当前实现里，这会被视为冲突。

### 5.5 派发目标怎么选

`派发目标` 支持：

- `自动`
- `活动编辑器`
- `页面主体`
- `窗口`

推荐理解如下：

#### 自动

默认推荐。

处理逻辑：

1. 如果能从 keymap 反查到稳定内置命令或插件命令，优先执行稳定路径
2. 如果这是编辑器命令，或者当前页面存在活动可编辑区，优先发给活动编辑器
3. 如果允许直接发到 `window`，则可能回退到 `window`
4. 最后回退到 `body`

适合：

- 你不确定它到底是编辑器命令还是页面命令
- 你希望插件自行尝试最合适的目标

#### 活动编辑器

会尽量找到当前激活的编辑器内容区，并向该区域派发 `keydown`。

适合：

- 加粗
- 斜体
- 标题级别
- 列表、表格等明显依赖编辑器焦点的命令

#### 页面主体

会向 `document.body` 派发事件。

适合：

- 更偏页面级、但不要求必须送到 `window` 的快捷键

#### 窗口

会向 `window` 派发事件。

适合：

- 某些只监听全局窗口键盘事件的能力

风险：

- 这通常比编辑器目标更脆弱
- 如果自动模式已经能工作，没必要强制改成 `窗口`

### 5.6 “先发 Escape”有什么用

开启后，会在真正发送快捷键前，先往窗口发送一次 `Escape`。

它通常用于：

- 关闭当前弹出菜单
- 退出某些临时状态
- 让焦点回到更容易接收快捷键的位置

适合场景：

- 上一次操作可能留下菜单、对话框、选项面板
- 某个快捷键只有在“界面干净”时才生效

注意：

- 如果快捷键已经被反查并走“稳定内置命令 / 插件命令”路径，这个选项不会介入那条稳定路径。

### 5.7 冲突检测

设置页会阻止不同按钮使用重复的实验快捷键。

例如：

- 按钮 A 已经使用 `Ctrl+B`
- 你给按钮 B 再录入 `Ctrl+B`

那么本次修改不会保存，并会提示：

```text
快捷键 Ctrl+B 已被按钮「已占用按钮」使用。
```

### 5.8 推荐示例

#### 示例 1：把“加粗”挂到按钮

适合：编辑器内格式化

配置建议：

| 字段 | 值 |
| --- | --- |
| 动作类型 | `实验：快捷键适配` |
| 快捷键 | `Ctrl+B` |
| 派发目标 | `活动编辑器` |
| 先发 Escape | `关闭` |
| 允许直接发到 window | `关闭` |

说明：

- 这是最典型的编辑器快捷键场景
- 如果当前思源 keymap 能反查出编辑器命令，插件会优先用语义判断
- 反查不到时，也会把事件发给活动编辑器

#### 示例 2：把“命令面板 / 搜索类快捷键”挂到按钮

适合：页面级命令

配置建议：

| 字段 | 值 |
| --- | --- |
| 动作类型 | `实验：快捷键适配` |
| 快捷键 | `Ctrl+P` |
| 派发目标 | `自动` |
| 先发 Escape | `关闭` |
| 允许直接发到 window | `关闭` |

说明：

- 这类快捷键很可能在思源 keymap 中能直接反查为内置全局命令
- 一旦反查成功，会优先走稳定命令执行，而不是模拟按键

#### 示例 3：需要先清理界面状态的快捷键

适合：容易被菜单或对话框拦截的动作

配置建议：

| 字段 | 值 |
| --- | --- |
| 动作类型 | `实验：快捷键适配` |
| 快捷键 | `Ctrl+Shift+P` |
| 派发目标 | `自动` |
| 先发 Escape | `开启` |
| 允许直接发到 window | `按需开启` |

## 6. 重点二：实验“点击序列”

### 6.1 这是什么

它会按你配置的步骤，逐步：

1. 查找目标元素
2. 点击该元素
3. 等待一小段时间
4. 再执行下一步

这适合处理“先打开一个菜单，再点菜单项”的链式动作。

### 6.2 如何配置

在右侧动作设置中，选择 `实验：点击序列` 后，可以看到：

- 点击步骤列表
- 新增步骤
- 删除步骤
- 每一步的等待与重试参数
- `失败即停止`

每一步都有以下字段：

| 字段 | 说明 |
| --- | --- |
| `选择器` | 用于查找目标元素 |
| `等待超时(ms)` | 单次等待元素出现的最大时间 |
| `重试次数` | 当前步骤查找失败后，再重试多少轮 |
| `重试间隔(ms)` | 每轮重试之间再等多久 |
| `步骤后延迟(ms)` | 本步点击成功后，进入下一步前额外等待多久 |

默认值：

| 字段 | 默认值 |
| --- | --- |
| `等待超时(ms)` | `5000` |
| `重试次数` | `2` |
| `重试间隔(ms)` | `300` |
| `步骤后延迟(ms)` | `200` |
| `失败即停止` | `true` |

### 6.3 选择器支持什么写法

当前实现支持三类写法。

#### 写法 1：简单标识符

例如：

```text
barSettings
```

这类写法会尝试匹配：

- 元素 `id`
- `data-id`
- `data-menu-id`
- `data-type`
- `data-action`
- 同名 class
- `<use href="#...">` / `<use xlink:href="#...">`

适合：

- 你已经知道某个原生按钮或菜单项的标识
- 目标元素本身有稳定属性名

#### 写法 2：文本匹配

以 `text:` 开头，例如：

```text
text:复制块引用
text:设置
```

匹配规则：

- 会扫描根节点下所有元素文本
- 文本会先做空白归一化
- 找到完全相等的文本后，自动向上寻找可点击元素

适合：

- 菜单项文本稳定
- 你没有合适的 id / data 属性可以用

注意：

- 这是“精确文本匹配”，不是模糊包含
- 如果界面文案会随语言变化，稳定性会下降

#### 写法 3：原始 CSS 选择器

例如：

```text
.b3-menu__item
[data-id="barSettings"]
.toolbar__item[data-type="doc"]
```

只要字符串看起来像 CSS 选择器，当前实现就会按 CSS 查询。

适合：

- 你已经通过开发者工具确认了 DOM 结构
- 需要更精确地锁定元素

注意：

- CSS 选择器越依赖层级和类名，越容易在思源升级后失效

### 6.4 元素是如何被点击的

找到元素后，执行器会优先尝试：

1. 直接调用 `element.click()`
2. 如果失败，再派发一个 `MouseEvent("click")`

因此只要目标节点或其邻近节点能被识别为可点击元素，就有机会成功。

### 6.5 查找失败时会发生什么

如果某一步失败：

- 会记录失败步骤编号
- 会提示失败的选择器
- 如果开启了 `失败即停止`，整个序列立即终止
- 如果关闭了 `失败即停止`，则继续尝试后续步骤，但最终返回值仍会标记为未完全成功

运行时提示示例：

```text
点击序列第 2 步失败：text:复制块引用
```

### 6.6 推荐示例

#### 示例 1：两步打开设置再点菜单项

配置目标：

1. 先点击设置按钮
2. 再点击文本为“复制块引用”的菜单项

配置建议：

| 步骤 | 选择器 | 等待超时 | 重试次数 | 重试间隔 | 步骤后延迟 |
| --- | --- | --- | --- | --- | --- |
| 1 | `barSettings` | `1000` | `0` | `0` | `150` |
| 2 | `text:复制块引用` | `1500` | `1` | `100` | `0` |

建议：

- `失败即停止` 设为 `开启`
- 第二步通常要给菜单一点渲染时间，所以第 1 步后建议留出 `100ms` 到 `300ms`

#### 示例 2：仅通过文本点击一个明显菜单项

适合：快速原型验证

| 步骤 | 选择器 |
| --- | --- |
| 1 | `text:设置` |

说明：

- 这是点击序列的默认回退示例
- 文案稳定时好用，但如果主题、语言或上下文变化，容易点到错误位置或找不到

#### 示例 3：使用 CSS 选择器点击某个工具栏按钮

| 步骤 | 选择器 |
| --- | --- |
| 1 | `[data-id="barSettings"]` |
| 2 | `.b3-menu__item` |

说明：

- 适合你已经用开发者工具确认了属性结构
- 生产使用前，建议把第 2 步改成更精确的选择器，避免点到第一个同类节点

### 6.7 点击序列编写建议

- 先用 1 步版本验证目标元素是否能找到
- 再逐步增加第 2 步、第 3 步
- 优先使用标识符或稳定 `data-*` 属性
- 文本匹配次之
- 复杂层级 CSS 选择器最后再用
- 菜单、弹层、异步面板之间通常要留 `100ms` 到 `300ms` 的步骤后延迟
- 如果界面加载慢，优先增大 `等待超时`，再考虑增加 `重试次数`

### 6.8 点击序列设值示例：切换语言到 English

```json
{
  "actionType": "experimental-click-sequence",
  "actionId": "lang",
  "experimentalClickSequence": {
    "stopOnFailure": true,
    "steps": [
      {
        "selector": "lang",
        "value": "en_US",
        "valueMode": "value",
        "timeoutMs": 1000,
        "retryCount": 0,
        "retryDelayMs": 0,
        "delayAfterMs": 0
      }
    ]
  }
}
```

- `select` 推荐优先按 `value` 匹配
- `input` 和 `textarea` 会直接写入 `value`
- 设值成功后会派发 `input` 和 `change`

## 7. 两种实验动作怎么选

可以按下面的思路判断：

### 7.1 选“快捷键适配”

当满足以下特征时优先考虑：

- 你已经知道目标快捷键
- 目标动作本来就可以通过键盘触发
- 它更像“命令”而不是“界面流程”
- 希望尽量保留与 keymap 的语义对应关系

典型例子：

- 加粗
- 打开命令面板
- 切换某类编辑器格式

### 7.2 选“点击序列”

当满足以下特征时优先考虑：

- 目标动作本身不是一个清晰快捷键
- 它必须经过菜单、弹层、工具栏等 UI 流程
- 你需要“先点 A，再点 B”

典型例子：

- 先展开某个菜单，再点某个菜单项
- 点击某个原生按钮后继续点击弹出项

### 7.3 不建议使用实验动作的情况

- 已经能用 `内置命令` 解决
- 已经能用 `插件命令` 解决
- 目标依赖高度不稳定的 DOM 结构
- 目标文案会频繁变化

## 8. 常见问题与排查

### 8.1 点了按钮后提示“实验快捷键适配当前不可用”

先检查：

1. 当前是不是桌面端
2. 思源版本是否至少为 `3.5.7`
3. `experimental.shortcutAdapter` 是否被关闭

### 8.2 点了按钮后提示“实验点击序列当前不可用”

先检查：

1. 当前是不是桌面端
2. 思源版本是否至少为 `3.5.7`
3. `experimental.clickSequenceAdapter` 是否被关闭

### 8.3 快捷键设置后没有效果

建议按顺序排查：

1. 把派发目标改成 `活动编辑器`，确认是否是焦点问题
2. 试试开启 `先发 Escape`
3. 确认这个快捷键在思源当前 keymap 中确实存在
4. 避免和其他实验快捷键按钮冲突
5. 如果本来就能用 `内置命令` 或 `插件命令`，改回稳定动作类型

### 8.4 点击序列第一步就失败

通常是选择器写法不对。

建议：

1. 先尝试最简单的 `text:文案`
2. 再尝试目标元素的 `id` 或 `data-id`
3. 最后再用 CSS 选择器

### 8.5 点击序列第二步失败，但手动操作没问题

通常是“第一步点完后，第二步目标还没渲染出来”。

建议：

1. 增大第 1 步的 `步骤后延迟`
2. 增大第 2 步的 `等待超时`
3. 适当增加第 2 步的 `重试次数`

### 8.6 文本选择器以前能用，升级后失效

常见原因：

- 界面文案改了
- 当前语言变了
- 目标元素不再是可点击节点，或可点击节点结构变了

建议：

- 改用 `data-*` 标识或更稳定的 CSS 选择器

## 9. 推荐实践

- 优先稳定动作，实验动作只解决稳定路径覆盖不到的场景
- 对实验快捷键，先从 `自动` 开始，不行再改 `活动编辑器`
- 对点击序列，先做最小可行链路，再逐步增加步骤
- 导出配置留档，方便回滚和跨设备迁移
- 思源升级后，优先回归测试所有依赖 DOM 的点击序列

```json
      }
    ]
  }
}
```
